## Issues Found

### Issue #1 — MAJOR: 에러 화면 경로에서 `on_loaded` 가 JS 브릿지 코드를 실행한다

**위치**: `main_webview.py` line 408–474 (`on_loaded`) + line 511–531 (실패 분기)

**문제**:
실패 분기(line 511–531)에서는 `_navigated[0] = True` 를 설정한 뒤 `window.load_html(error_html)` 을 호출한다. 그러면 PyWebView 의 `loaded` 이벤트가 다시 발화되고, `on_loaded` 가 호출된다. 이때 `_navigated[0]`이 이미 True 이므로 **`on_loaded`는 2차 분기로 진입해 메인 URL용 JS 브릿지(`evaluate_js`로 fetch + window.SQM_API_BASE 등)를 에러 페이지에 주입한다.**

에러 페이지는 정적 HTML이고 SQM_API_BASE / 에러 후크가 필요하지 않다. 더 나쁜 점: 그 JS 안의 `fetch(window.SQM_API_BASE + "/api/log/frontend-error", …)` 가 죽어 있는 백엔드(API 서버 시작 실패가 바로 이 분기 진입 조건)에 주기적으로 요청을 시도하다 실패해 콘솔 잡음을 만든다. 또한 사용자가 에러 화면을 보는 동안 `console.error` 후크 설치 등은 의미가 없다.

**1차(Main Ruby)가 놓친 이유**: `_navigated`를 단순히 "스플래시 vs 메인" 2-state 로 간주하고, 실패 경로의 load_html 도 똑같이 "메인 페이지 로드" 로 처리한 결과.

**Severity**: MAJOR (기능 동작은 하지만 의도와 다른 부수 효과 + 죽은 백엔드로 반복 fetch 요청이 발생).

**권장 수정**:
- 상태 플래그를 3-state 로 분리: `_page = ['splash' | 'main' | 'error']`
  - 또는 `_navigated[0]` 와 별도로 `_is_error = [False]` 추가
- `on_loaded`에서:
  ```python
  if not _navigated[0]:        # splash
      ... maximize 처리 ...
      return
  if _is_error[0]:             # error 화면
      return                    # JS 브릿지 미설치
  # 메인 URL 만 JS 브릿지 설치
  window.evaluate_js(f''' ... ''')
  ```
- 실패 분기에서 `_is_error[0] = True; _navigated[0] = True; window.load_html(error_html)` 순으로 설정.

### Issue #2 — MINOR: `loaded` 이벤트와 `webview.start` 콜백 사이의 잠재적 race

**위치**: line 408–474 (`on_loaded`) vs line 504–508 (`on_window_started` 의 load_url 호출)

**문제**:
PyWebView 구현체에 따라 첫 번째 `loaded` 이벤트(스플래시 HTML 로드 완료)가 `on_window_started` 보다 **늦게** 발화될 수 있다. 만약:

1. `on_window_started` 가 `wait_for_api` 를 매우 빨리 (예: 100ms) 통과하고
2. `_navigated[0] = True` 설정 + `window.load_url(url)` 을 호출하는데
3. 그 시점에도 첫 번째 splash `loaded` 이벤트가 아직 발화되지 않았다면

→ 처음 발화되는 `loaded` 이벤트는 _navigated[0] 이미 True 이므로 곧바로 2차 분기(메인 URL JS 브릿지)에 진입한다. 그런데 그 시점에서 DOM 은 아직 메인 페이지가 아니라 splash 또는 main 의 부분 로드 상태일 수 있어 `evaluate_js` 가 빈 document 에 대해 실행된다.

**Severity**: MINOR (실측에서는 wait_for_api 가 보통 1초 이상 걸려 거의 발생하지 않지만, 캐시·Hot start 환경에서 0.3초만에 통과 가능).

**권장 수정**:
- `_navigated`를 카운터(`_load_count = [0]`)로 바꾸고 `on_loaded`에서 카운트 증가 후 분기.
  - 1번째 발화: splash maximize 처리만
  - 2번째 발화: 메인 URL JS 브릿지 설치
  - 3번째+: 무시 (또는 `_is_error` 분기 후 JS 브릿지 미설치)
- 또는 `on_window_started`에서 `wait_for_api`가 1초 미만에 끝나면 잠깐 더 대기하여 splash loaded 이벤트가 안정적으로 먼저 발화되도록 보장.

### Issue #3 — MINOR: `window.maximize()` 호출이 splash 단계에 묶여 있어 메인 URL 로드 후 풀릴 가능성

**위치**: line 411–416 (`on_loaded` 1차 분기)

**문제**:
최대화 복원을 splash 페이지가 로드될 때 한 번 수행하고 끝낸다. 일부 PyWebView/WinForms 구현은 `load_url` 이후에 윈도우 사이즈가 재조정되며 최대화 상태를 잃을 수 있다. 메인 URL 로드 후에도 다시 한 번 maximize 를 시도하는 것이 안전하다(또는 적어도 `_win_max` 가 여전히 True 면 메인 페이지에서 한 번 더 보장).

**Severity**: MINOR (재현 의존, 환경에 따라 정상 동작).

**권장 수정**:
- 2차 분기(메인 URL 로드 완료) 진입 시 `_win_max` 가 True 면 `window.maximize()` 한 번 더 호출.

### Issue #4 — MINOR: API 시작 실패 시 backend_thread 가 즉시 종료되어 리소스 회수 불가

**위치**: line 270–287

- 또는 `wait_for_api` 안에서 `not backend_thread.is_alive()` 시 조기 반환.

### Issue #5 — NIT: `on_closing` 이 `webview` 종료 직전 호출되는데 `os._exit(0)` 가 그 직후 실행됨

**위치**: line 537–540

**문제**:
`webview.start()` 가 반환된 직후 `os._exit(0)` 를 호출한다. 그러나 atexit / Python finalizers / FastAPI shutdown event 가 호출되지 않는다(이는 의도된 디자인이라고 코멘트도 달려 있음). `on_closing` 이 동기 호출이라 `window.width`/`height` 저장은 보장되지만, 만약 `on_closing` 안의 `save_window_state` 가 디스크 flush 전에 `os._exit` 가 발생하면 잘려 쓰기 가능성이 이론상 존재한다 — 단, `json.dump` 는 close() 시 flush 되고 `with open` block 이 빠져나간 시점에는 OS 캐시에 들어가 있어 거의 무해.

**Severity**: NIT.

**권장 수정**: 굳이 없음. 의도된 설계. 다만 현재 주석("FastAPI daemon 스레드가 살아있어도 깔끔하게 종료") 옆에 "`on_closing` 은 webview.start 반환 전에 동기 호출됨" 한 줄 추가하면 차후 유지보수에 도움.

## Code Snippet — Issue #1 의 핵심 결함 위치

```python
# line 405–407
# [P1 PATCH] on_loaded gating: 첫 호출 = splash, 두 번째 호출부터 = real URL (JS 브릿지 설치)
_navigated = [False]

def on_loaded():
    if not _navigated[0]:
        # 1차 발화: 스플래시
        if _win_max:
            window.maximize()
        return
    # ↓↓↓ 여기로 에러 페이지 로드 시에도 진입함! ↓↓↓
    window.evaluate_js(f'''
        window.SQM_API_BASE = "http://{API_HOST}:{API_PORT}";
        ...
        fetch(window.SQM_API_BASE + "/api/log/frontend-error", ...)
        ...
    ''')

# line 511–529 — 실패 분기
log.error("API 시작 실패 -> 오류 화면 표시")
_navigated[0] = True            # ← 이걸로 인해 위 on_loaded 가 메인 분기로 진입
error_html = '...'
window.load_html(error_html)    # ← 이게 loaded 이벤트 트리거 → on_loaded() → JS 브릿지 주입
```
