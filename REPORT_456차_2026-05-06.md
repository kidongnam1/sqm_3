## 🎯 6-Pass 검증의 핵심 교훈

### Lesson 1 — 검증 차수당 ROI 압도적
- 1차 자체 평가: 30~60% 신뢰도
- +2차: MAJOR 1 + MEDIUM 1 발견 (85%)
- +3차: **CRITICAL 1 발견** (95%) — production 사고 사전 방지
- +4차: 운영 리스크 3건 식별 (98%)
- +5차: 시스템 갭 9건 + JS 8 파일 스캔 (99%)
- +6차: Final ship 게이트 통과

→ **`templates/three-pass-verification.md` SOP 자산화 완료** (다음 프로젝트 즉시 재사용)

### Lesson 2 — JS 이벤트 핸들러 = idempotent 필수
- `console.error` wrap, `addEventListener` 등은 가드 없이 재진입 시 무한 재귀 + 누적
- 5차 JS 스캔 결과: SQM 프론트엔드 8 파일에 41+ 무가드 listener 발견
- → **`templates/js-reentry-guard.md` 패턴 표준화 완료**

### Lesson 3 — Sub-agent 인스턴스 분리 = 핵심
- 같은 인스턴스가 자기 코드 검증 시 confirmation bias
- 1차 (Sub A~E) → 2차 (Sub F~J) → 3차 (Sub K~O) → 4차 (Sub K2~L2) → 5차 (Sub M~N) → 6차 (Sub O~P) 처럼 차수마다 인스턴스 advance 필수
- 다양한 관점 (race, integration, holistic, JS scan, pre-deploy) 각자 fresh 분석

### Lesson 4 — 운영 리스크는 stress test 로만 발굴
- 1-3차는 코드 정합성 위주 검증
- 4차 stress 시나리오 추가로 **double-instance race**, **백엔드 silent fail** 등 운영 환경 위협 식별
- → 향후 모든 패치에 stress test 차수 의무화 권장

### Lesson 5 — 시스템 검증 ≠ 단위 검증
- 1-3차는 파일별 검증
- 5차 holistic 에서 thread budget, memory, 로깅, 테스트 갭 등 emergent property 검출
- 시스템 시각 검증을 별도 차수로 두는 것이 가치 있음
