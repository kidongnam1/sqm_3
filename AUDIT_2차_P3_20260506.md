## Issues Found

### 🟡 MEDIUM-1 — No `atexit` shutdown for `_db_executor`
The executor uses non-daemon worker threads (Python default). On interpreter
exit, Python's own `concurrent.futures` atexit hook will block until in-flight
DB tasks finish. For a Tk/PyQt UI app, a long-running locked SQLite call could
delay shutdown indefinitely.

**Recommended fix:**
```python
import atexit
atexit.register(_db_executor.shutdown, wait=False)
```
Place immediately after line 865.

### 🟡 MEDIUM-2 — Caller cannot distinguish success-with-Exception-result from failure
`payload = exc if exc is not None else f.result()` collapses both cases into
one argument. If a callable legitimately *returns* an Exception instance (rare
but legal — e.g., a validator returning the exception), `on_done` cannot tell
it apart from a raised exception.

**Recommended fix:** Use a 2-arg signature `on_done(result, exc)` or pass a
`Future` to the callback so callers can call `.exception()` themselves. Low
priority — current pattern is conventional.

### 🟢 LOW-1 — Docstring says "follow-up thread" but doesn't name it
Lines 861-863 say `on_done(result)` is invoked "from a follow-up thread."
In practice, `add_done_callback` runs the callback on the *worker thread*
(or the calling thread if already done). The 1차 docstring claims worker —
matches CPython behavior. Suggest tightening the module-level comment to
match: "invoked on the worker thread (or caller's thread if already done)."

### 🟢 LOW-2 — `db_query_async` only wraps `fetchall`
No `db_query_one_async` for `fetchone`. Acceptable for now; callers can use
`db_execute_async(db.fetchone, sql, params, on_done=cb)` directly.

## Recommended Fixes (priority order)

1. **MEDIUM-1** — Add `atexit.register(_db_executor.shutdown, wait=False)`
   right after the executor definition. One-line change, prevents shutdown hang.
2. **LOW-1** — Tighten module-level docstring: replace "follow-up thread" with
   "worker thread (or caller's thread if the future is already done)."
3. **MEDIUM-2** — (Optional) Consider `on_done(result, exc)` signature for
   future API clarity. Defer until a real caller hits the ambiguity.
4. **LOW-2** — (Optional) Add `db_query_one_async` if/when `fetchone` becomes
   a hot path in async-converted UI handlers.
