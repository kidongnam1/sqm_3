# v866 재고 추이 차트 검토 및 정상화 작업

## 1. 현재 상태

v866 웹에는 `재고 추이 차트` 메뉴와 API 연결이 존재한다.

- 메뉴: `재고 > 재고 추이 차트`
- 프론트 액션: `onStockTrendChart`
- 연결 API: `/api/q/inventory-trend`
- 백엔드 함수: `backend/api/queries.py`의 `get_inventory_trend()`

하지만 현재 기준으로는 정상 작동한다고 보기 어렵다.

## 2. 현재 문제

### 2.1 `inventory_snapshot` 데이터가 없음

현재 DB 확인 결과는 다음과 같았다.

- `inventory` rows: 20
- `stock_movement` rows: 20
- `inventory_snapshot` rows: 0

`/api/q/inventory-trend`는 `inventory_snapshot` 테이블을 조회한다.

따라서 `inventory_snapshot`이 비어 있으면 메뉴를 눌러도 실제 추이 데이터가 나오지 않는다.

### 2.2 프론트가 차트로 렌더링하지 않음

현재 프론트의 `onStockTrendChart`는 일반 JSON 조회 모달로 연결되어 있다.

API가 `chart.labels`, `chart.series` 형태의 데이터를 반환하더라도,
프론트에는 이를 전용 차트나 보기 좋은 추이 표로 그리는 전용 모달이 없다.

즉, 현재는 "차트"라기보다는 일반 데이터 조회에 가까운 상태다.

## 3. 정상 작동을 위한 필수 작업

아래 작업을 하면 v866 웹의 재고 추이 메뉴는 실사용 가능한 수준으로 정상화된다.

### 3.1 현재 재고 기준으로 `inventory_snapshot` 생성

현재 `inventory`, `inventory_tonbag` 기준으로 오늘 날짜의 스냅샷을 생성해야 한다.

스냅샷에 들어갈 대표 항목은 다음과 같다.

- `snapshot_date`
- `total_lots`
- `total_tonbags`
- `total_weight_kg`
- `available_weight_kg`
- `picked_weight_kg`
- `product_summary`
- `created_at`

### 3.2 앱 시작 또는 일 단위 자동 저장

앱이 시작될 때 오늘 날짜의 스냅샷이 없으면 자동 생성하도록 해야 한다.

권장 방식:

- 프로그램 시작 시 1회 확인
- 오늘 날짜 스냅샷이 없으면 생성
- 이미 있으면 중복 생성하지 않음

이렇게 하면 사용자가 별도 버튼을 누르지 않아도 매일 재고 추이가 누적된다.

### 3.3 `/api/q/inventory-trend` 안정화

현재 API는 `inventory_snapshot`을 조회하는 구조다.

필요한 보완:

- 데이터가 없을 때 명확한 안내 메시지 반환
- 데이터가 있을 때 날짜 오름차순 반환
- 총재고, 판매가능, 출고/피킹 중량을 MT 단위로 반환

### 3.4 프론트 전용 모달 추가

`onStockTrendChart`를 일반 JSON 모달이 아니라 전용 모달로 연결해야 한다.

전용 모달은 다음 상태를 처리해야 한다.

- 데이터 없음
- 데이터 1건
- 데이터 여러 건
- API 오류

### 3.5 데이터 없을 때 안내 메시지 표시

`inventory_snapshot`이 비어 있으면 사용자에게 다음 취지의 메시지를 보여줘야 한다.

> 스냅샷 데이터가 아직 없습니다. 오늘 기준 스냅샷을 생성한 뒤부터 매일 누적됩니다.

가능하면 "오늘 스냅샷 생성" 버튼도 제공할 수 있다.

### 3.6 데이터 있을 때 표 또는 간단 차트 표시

최소 구현은 표 형태다.

표 컬럼 예시:

- 날짜
- LOT 수
- 톤백 수
- 총재고(MT)
- 판매가능(MT)
- 피킹/출고대기(MT)

추가로 간단한 막대 차트 또는 선형 차트를 붙이면 더 좋다.

## 4. 위 작업만 하면 정상 작동하나?

결론적으로, 위 작업만 해도 v866 웹의 `재고 추이 차트`는 정상 작동 범위에 들어간다.

정상 작동의 의미는 다음과 같다.

- 메뉴를 눌렀을 때 빈 화면이 아니라 의미 있는 화면이 나온다.
- 현재 재고 기준 스냅샷이 생성된다.
- 이후 매일 스냅샷이 쌓인다.
- 쌓인 스냅샷을 날짜별로 볼 수 있다.
- 사용자가 총재고, 판매가능 재고, 피킹/출고대기 재고 흐름을 확인할 수 있다.

## 5. 주의할 점

`inventory_snapshot`은 과거 데이터를 자동으로 복원하지 않는다.

오늘 처음 스냅샷을 만들면 오늘 기준 1개 지점만 생긴다.

따라서 첫날에는 다음처럼 보일 수 있다.

- 표 1행
- 차트 점 1개
- 추이선이 거의 보이지 않음

이것은 오류가 아니라 스냅샷 누적 방식의 자연스러운 결과다.

시간이 지나면서 매일 저장되어야 실제 추이 차트가 된다.

## 6. `stock_movement` 기반 과거 추정 추이

더 고급 방식으로는 `stock_movement` 테이블을 사용해 과거 추이를 추정할 수 있다.

현재 DB에는 `stock_movement` 데이터가 존재한다.

이 데이터를 사용하면 다음과 같은 방식으로 과거 추이를 재구성할 수 있다.

- 현재 재고를 기준점으로 잡음
- 과거의 입고/출고/이동 이력을 역산
- 날짜별 예상 재고량을 계산
- `inventory_snapshot`이 없던 기간의 추정 추이를 생성

하지만 이 방식은 단순 스냅샷 생성보다 훨씬 복잡하다.

주의할 점:

- 모든 입출고 이력이 `stock_movement`에 완전하게 기록되어 있어야 한다.
- 이동, 반품, 샘플, 출고 취소, 정합성 복구 이력을 어떻게 해석할지 정해야 한다.
- 누락된 movement가 있으면 과거 추정값이 틀어질 수 있다.
- 추정 데이터와 실제 스냅샷 데이터를 구분해야 한다.

따라서 `stock_movement` 기반 과거 추정 추이는 별도 고급 작업으로 보는 것이 맞다.

## 7. 권장 순서

1. 현재 재고 기준 `inventory_snapshot` 생성 기능 추가
2. 앱 시작 또는 일 단위 자동 스냅샷 저장
3. 프론트에 `inventory-trend` 전용 모달 추가
4. 데이터 없음/1건/여러 건 상태 처리
5. 표 또는 간단 차트 표시
6. 필요 시 별도 단계로 `stock_movement` 기반 과거 추정 추이 구현

## 8. 결론

우선 필수 작업만 해도 v866의 재고 추이 메뉴는 실사용 가능한 수준으로 정상화된다.

다만 과거 30일 추이를 바로 보고 싶다면 `stock_movement` 기반 재구성 작업이 추가로 필요하다.

이 작업은 데이터 해석 규칙이 필요하므로 기본 정상화 작업과 분리해서 진행하는 것이 안전하다.
