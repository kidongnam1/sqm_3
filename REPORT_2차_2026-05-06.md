## 🔴 핵심 발견 — P1 MAJOR 이슈 (즉시 수정됨)

### Issue #1 (MAJOR) — `_navigated` 2-state 플래그의 위험

**문제**:
```python
_navigated = [False]   # 1차 코드 — 2-state 플래그
```
- `_navigated[0] = True` 가 **메인 URL 성공** + **오류 화면 표시** 양쪽 모두에서 설정됨
- 결과: 오류 화면 (`window.load_html(error_html)`) 이 `loaded` 이벤트를 발화시키면 `on_loaded` 가 메인 URL 분기로 진입
- 메인 URL 분기 = JS 브릿지 설치 (`fetch("/api/log/frontend-error", ...)`)
- 오류 진입 전제 조건 = 백엔드 죽음 → **죽은 백엔드에 fetch 무한 시도** ❌

**영향도**: API 시작 실패 시 (예: 포트 충돌, uvicorn 크래시) 네트워크 무한 로그 시도

**즉시 수정** (이 보고서 작성 직전):
```python
# 2차 수정 — 3-state 플래그
_phase = ["splash"]  # "splash" | "main" | "error"

def on_loaded():
    if _phase[0] == "splash":
        # 최대화 복원만
        return
    if _phase[0] == "error":
        # JS 브릿지 비설치 (죽은 백엔드 fetch 루프 방지)
        log.info('on_loaded: error 페이지 — JS 브릿지 비설치')
        return
    # _phase[0] == "main"
    # JS 브릿지 + 에러 후크 설치
```

**상태**: ✅ 적용 완료. 검증: `_phase[0] = "main"` (성공) vs `_phase[0] = "error"` (실패) 분리됨.

## 🟡 P3 MEDIUM 이슈 (즉시 수정됨)

### MEDIUM-1 — `atexit.register(_db_executor.shutdown)` 누락

**문제**:
- `_db_executor = ThreadPoolExecutor(max_workers=4, ...)` 정의됨
- 그러나 **인터프리터 종료 시 정리 메커니즘 없음**
- 만약 DB 락에 걸린 작업이 워커 스레드에 있으면, 인터프리터 종료가 블록될 수 있음

**즉시 수정**:
```python
import atexit as _atexit
_atexit.register(_db_executor.shutdown, wait=False)
```
(`engine_modules/database.py` line 866-869 영역)

**상태**: ✅ 적용 완료.

---

### MEDIUM-2 — `on_done` 시그니처 모호성

**문제**:
- 현재 `on_done(payload)` 에서 payload 가 **결과** 또는 **예외 객체**
- 호출자는 `isinstance(payload, Exception)` 으로 분기해야 함
- 만약 콜러블이 정당하게 `Exception` 객체를 반환하면 분기 불가

**제안 (수정 안 함, 향후 검토)**:
```python
on_done(result, exc)  # 항상 두 인자, 둘 중 하나는 None
```
- 변경하려면 db_query_async / db_execute_async 양쪽 시그니처 + 호출자 수정 필요
- 현재 호출자 0건 → 향후 첫 호출자 도입 시점에 함께 결정

**상태**: 🟡 향후 검토 (변경 안 함).

## 🎯 핵심 교훈 (Lessons Learned)

| Insight | 적용처 |
|---|---|
| **2차 독립 재검증의 ROI는 압도적** | 30분 추가 → MAJOR 1건 + MEDIUM 2건 검출. 1차만 했으면 프로덕션에서 발견됐을 것 |
