## Critical Findings

### 🔴 [CRIT-1] `console.error` 후크 무한 재귀 (line 470~480)
**증상**
- `_phase[0] == "main"` 상태에서 `on_loaded` 가 2회 이상 발화하면
  `var _origErr = console.error;` 가 **이미 래핑된 함수**를 다시 캡처한다.
- 결과: `console.error` 호출 → 새 래퍼 → 이전 래퍼(이전 _origErr) → ... → 스택오버플로우.

**재현 트리거**
1. SPA 내 라우팅이 `pushState` 가 아니라 실제 페이지 로드를 발생시키는 경우.
2. 사용자가 `<a href>` 클릭으로 동일 origin 내비게이션.
3. `window.load_url` 자체가 success path 에서 on_loaded 를 1번 더 발생시킴 — **즉 거의 항상 1번은 중첩됨.**

**증거**
- line 470: `var _origErr = console.error;` — 매 호출마다 캐치, IIFE 가 재호출 방지 가드 없음.
- line 446 `addEventListener("error", ...)` 도 매번 추가됨 → 같은 에러가 N회 백엔드로 POST.
- line 459 `unhandledrejection` 동일 문제.

**핫픽스 (필수)**
```javascript
(function installErrorBridge() {{
    if (window.__SQM_BRIDGE_INSTALLED__) return;   // 가드 추가
    window.__SQM_BRIDGE_INSTALLED__ = true;
    // ...기존 코드...
}})();
```

### 🟡 [HIGH-2] Splash → Main 경합 (race window 존재 확인)
- `wait_for_api` 가 ~100ms 미만에 반환하면 `_phase[0] = "main"` 이
  splash 의 `on_loaded` 발화보다 **먼저** 일어날 수 있음.
- 결과: splash 페이지에서 main 분기로 진입 → 무의미한 `evaluate_js` (`SQM_API_BASE` 설정만 발생, 시각적 영향 無).
- 이어서 `load_url` 이 main on_loaded 를 트리거 → 브릿지 **2회 설치** → CRIT-1 재귀 폭발로 직결.
- **상기 [CRIT-1] 핫픽스로 동시 해결됨.**

---

### 🟡 [MED-3] `_phase[0]` 멀티스레드 가시성
- `on_window_started` 는 webview 콜백 스레드, `on_loaded` 는 GUI 스레드.
- CPython GIL 로 list slot 1개 쓰기는 atomic 하지만 **happens-before 보장 없음**.
- 실무상 PyWebView 의 이벤트 디스패치가 OS 메시지 큐를 거치며 implicit barrier 가 들어가 거의 안전.
- **Acceptable risk** — 하지만 권장: `_phase` 를 `threading.Event` 두 개(`api_ready`, `api_failed`) 로 바꾸면 Lock 없이도 명시적.

---

### 🟢 [LOW-4] `load_url` 실패 후 `_phase` 미복원 (line 517-519)
- `_phase[0] = "main"` 이미 설정 후 `load_url` 예외 → 후속 on_loaded 가 main 분기로 진입.
- 현실적으로 `load_url` 실패는 윈도우 파괴 시점뿐이라 후속 on_loaded 가 거의 없음.
- 보강 필요 시: `except` 절에서 `_phase[0] = "error"` + `load_html(error_html)` 폴백.

---

### 🟢 [LOW-5] 백엔드 import 실패 시 10초 freeze (line 270-282)
- `backend.api` import 실패 → `run_api_server` 가 즉시 return → `wait_for_api` 가 10s 풀 타임아웃.
- UX 개선안: `_setup_backend` 가 `threading.Event(api_failed)` 를 set 하고, `wait_for_api` 가 이벤트도 watch 하도록 변경.

## Recommended Hot-Fix (Priority Order)

1. **즉시** — JS IIFE 상단에 `if (window.__SQM_BRIDGE_INSTALLED__) return;` 가드 추가 (CRIT-1).
2. **선택** — `console.error` 후크에 `if (console.error.__sqmHooked) return;` 보강.
3. **선택** — `_phase` 를 `threading.Event` 기반으로 재설계 (MED-3 + LOW-5 동시 해결).

**[CRIT-1] 가드 한 줄을 추가하지 않으면 메인 화면에서 첫 console.error 가 발생하는 순간 frontend freeze + RecursionError 백엔드 폭주가 발생한다. 이 패치 없이는 production 배포 불가.**
